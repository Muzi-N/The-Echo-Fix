# ECHO API

Backend API for ECHO, a modern messaging platform. Milestone one delivers a deployable foundation: authentication, users, contacts, one-to-one conversations, and real-time messaging with delivery and read receipts, typing indicators, and presence.

This service is built to deploy to Render with managed PostgreSQL and Redis, and to be consumed by a mobile client (built separately in a later milestone).

## Stack

NestJS 10 on Node 22, PostgreSQL 16 via TypeORM, Redis 7 via ioredis, Socket.IO for real-time transport, JWT access tokens with rotating refresh tokens, argon2id password hashing, class-validator for input validation, Swagger for API docs.

## What is included in milestone one

Authentication covering registration, phone OTP verification, login, refresh-token rotation, single-device logout, and password reset. User profiles with privacy controls for last-seen visibility and read receipts. Contacts with aliases, favorites, and blocking. Direct conversations with get-or-create semantics. Messaging with send, list (keyset paginated), edit, soft delete, and search. Real-time delivery over WebSocket with handshake JWT auth, presence tracking across instances, typing indicators, and delivered/read receipt propagation. A health endpoint that probes the database and Redis. Global rate limiting, security headers, URI versioning, and a uniform error envelope.

Deferred to later milestones: groups, channels, communities, stories, voice and video calling, the media upload pipeline, end-to-end encryption, AI features, and the admin panel. The schema and module boundaries were chosen so these can be added without rework.

## Local development

Prerequisites: Node 22, a running PostgreSQL 16, and a running Redis 7.

```bash
npm install
cp .env.example .env        # then edit DATABASE_URL and the JWT secrets
npm run migration:run       # create the schema
npm run seed                # optional: two demo users
npm run start:dev           # watch mode on http://localhost:3000
```

API docs are served at `http://localhost:3000/api/docs`. All routes are under `/api/v1`.

## Environment variables

See `.env.example` for the full list. Required in every environment: `DATABASE_URL`, `JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET`. The app validates its environment on boot and refuses to start if a required variable is missing or malformed.

## REST surface (v1)

Auth: `POST /auth/register`, `POST /auth/verify`, `POST /auth/login`, `POST /auth/refresh`, `POST /auth/logout`, `POST /auth/password-reset/request`, `POST /auth/password-reset/confirm`.

Users: `GET /users/me`, `PATCH /users/me`, `PATCH /users/me/privacy`, `DELETE /users/me`, `GET /users/search?q=`, `GET /users/:id`.

Contacts: `POST /contacts`, `GET /contacts`, `PATCH /contacts/:id`, `DELETE /contacts/:id`.

Conversations: `POST /conversations/direct`, `GET /conversations`, `GET /conversations/:id`.

Messages: `GET /conversations/:id/messages?limit=&before=`, `GET /messages/search?q=`, `PATCH /messages/:id`, `DELETE /messages/:id`.

Health: `GET /health`.

Every route except auth and health requires a `Authorization: Bearer <accessToken>` header.

## Real-time protocol

Connect a Socket.IO client to the service root, passing the access token in the handshake auth payload:

```js
const socket = io(BASE_URL, { auth: { token: accessToken } });
```

The connection is rejected if the token is missing or invalid. Inbound events from the client: `conversation:join`, `message:send`, `message:read`, `typing:start`, `typing:stop`. Outbound events to the client: `message:new`, `message:status`, `typing:update`, `presence:update`, `error`.

Events fan out through Redis pub/sub, so delivery is correct when more than one API instance runs behind a load balancer.

## Deploying to Render

The repository includes `render.yaml`, a blueprint that provisions the API web service, a managed PostgreSQL 16 database, and a managed Redis instance, and wires their connection strings into the service automatically. JWT secrets are generated by Render on first deploy.

1. Push this repository to GitHub.
2. In Render, create a new Blueprint and point it at the repository.
3. Render reads `render.yaml`, provisions the resources, runs the migration via the pre-deploy command, then starts the service.
4. The health check at `/api/v1/health` gates the service going live.


## Security notes

Passwords are hashed with argon2id and never returned. Refresh tokens are stored only as SHA-256 hashes, one row per device, and are rotated on every refresh so a stolen token is single-use. JWT auth is applied globally and fails closed; routes opt out explicitly. Input is strictly validated and unknown fields are rejected. Helmet sets security headers and a global rate limit is in place. See `SECURITY.md` for the full model.
